# DECISIONS

The requirements PDF + README API contract leave several details to the
implementer. Each row is one ambiguity, the resolution shipped in this
submission, and the one-line reason. Every choice is defensible in interview
and reflected in `plan.md`, the code, and the tests.

| # | Ambiguity | Resolution | Rationale |
|---|-----------|------------|-----------|
| 1 | **JWT logout strategy** — stateless expiry vs server-side invalidation? | **Deny-list keyed by `jti`** (token_denylist table) + an hourly `@Scheduled` purge of expired rows. | A stateless token can't be "logged out" — the README explicitly requires `/auth/logout` to invalidate the current token, which forces server-side state. Deny-list is the minimal addition. |
| 2 | **Concurrency mechanism** — pessimistic row locks vs optimistic versioning? | **Optimistic** via JPA `@Version` on `Ticket` and `Comment`. Conflicts surface as `409 Conflict` through `GlobalExceptionHandler`. | Pessimistic locking serializes contention and complicates the transactional boundary. Optimistic is the JPA-native default, low-overhead, and correct for the stated rule ("no simultaneous edits"). |
| 3 | **Optimistic version transport** — does the client send `version` in the PATCH body? | **No** — `UpdateTicketRequest` does not expose `version`. The DB-level `@Version` still rejects true overlapping transactions. | Strictly respects the README contract (the documented PATCH body has no `version` field). Trade-off: stale-read followed by sequential write would not be caught — accepted, with a note in `plan.md`. |
| 4 | **Status transition strictness** — allow skipping (TODO→DONE)? Backward moves (DONE→TODO)? | **Strict one-step forward only**: TODO→IN_PROGRESS→IN_REVIEW→DONE. Anything else → `409`. | The requirement's literal reading ("can only move forward, no skipping"). Encoded once in `TicketStatusTransitions` and exhaustively tested via `TicketStatusTransitionsTest`. |
| 5 | **DONE-immutability scope** — only block status changes, or all fields? | **All fields blocked** once `status == DONE`. | The requirement says a DONE ticket "can no longer be updated" — interpreting that as status-only would let priority, assignee, and title drift after closure. Stricter reading is safer. |
| 6 | **Escalation cadence** — how often does the scheduler run? | **Every minute** (`cron = "0 * * * * *"`). | Most aggressive cadence that still keeps the audit log signal-to-noise sane. The rule is idempotent, so over-running is harmless. |
| 7 | **"DEVELOPER in the project"** for auto-assignment / workload — only project-members, or every DEVELOPER user? | **Every DEVELOPER user** is a candidate; no project-membership concept. | The schema has no project_members table and the README does not specify one. Workload counts are scoped to the project, so cross-project load is invisible. |
| 8 | **Create-status policy** — must a new ticket start as TODO? | **Accept any status verbatim**. | The README's create body includes `status` as a free-form field. Forcing TODO would surprise an API consumer following the published contract. |
| 9 | **Auto-assign tie-break** — when two developers have the same open-ticket count, who wins? | **Lowest `users.id`** (registration order). | Deterministic, observable, and avoids hidden RNG. Encoded in the `findDeveloperWorkload` SQL `ORDER BY count ASC, u.id ASC`. |
| 10 | **Attachment storage location** — filesystem, S3, or DB? | **Postgres `BYTEA` column** on `attachments.data`. | No filesystem dependency, transactional with the rest of the row, fine within the 10 MB cap. Easy to swap for an object store later (the `AttachmentService` interface stays). |
| 11 | **Attachment whitelist enforcement** — content-type header alone, or also payload sniffing? | **Both** — header must be in the whitelist (`image/png`, `image/jpeg`, `application/pdf`, `text/plain`) AND magic-byte signature (or strict UTF-8 for text) must match. | A client lying about the content type would slip past a header-only check. Magic-byte sniffing is cheap and catches it. |
| 12 | **CSV partial-success model** — atomic ("all or nothing") or per-row? | **Per-row transactions** via a separate `TicketService.create(...)` call. The import returns `{created, failed, errors[]}`. | The README explicitly publishes the partial-success response shape. Atomic import would be incompatible with that contract. |
| 13 | **Unknown @mentions** — error or silent drop? | **Silent drop** in `CommentService.resolveMentions`. | The README does not specify; treating typos as errors would break ergonomic comment writing. Documented as a known behavior — unknown handles are simply not stored in `comment_mentions`. |
| 14 | **DELETE /users with active projects** — cascade, soft-delete the projects, or refuse? | **Refuse** with `409 Conflict`: `"Cannot delete user: owns N project(s). Reassign first."` | Cascading would silently destroy project state; soft-deleting projects without a user-initiated step would surprise an admin. Forcing reassignment is the safest UX. |
| 15 | **HTTP status for illegal state transitions** (DONE-immutable, blocked-by-dependency, skipping a step) | **`409 Conflict`** uniformly, via `ConflictException` → `GlobalExceptionHandler`. | All three are "the request is well-formed but conflicts with the current state of the resource" — exactly RFC 7231's definition of 409. Tested by `TicketIntegrationTest` and `DependencyIntegrationTest`. |
| 16 | **Soft-delete UX** — should the deleted row leak through normal GETs? | **No** — Hibernate `@SQLRestriction("deleted_at IS NULL")` filters every standard read; `GET /projects/deleted` and `/restore` use native SQL to bypass the filter and are gated by `@PreAuthorize("hasRole('ADMIN')")`. | Lets services stay oblivious to soft-delete in 95% of code paths while still exposing the admin recovery flow the README requires. |

Each row is testable: every resolution above either has a dedicated integration
test or is exercised by the contract walk in `run.md`.
